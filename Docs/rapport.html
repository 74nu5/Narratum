Narratum — état des lieux & évolutions

Le port des fonctionnalités narratives de StoryMaker est terminé, et l'interface a été entièrement refondue en un langage « éditorial » maison. Ce rapport dresse l'inventaire de ce qui a été livré, puis détaille les prochaines évolutions fonctionnelles.

Projet O:\Perso\Narratum Branche main Date 19/07/2026 Statut Fonctionnalités livrées · IHM refondue · roadmap ouverte
01

Résumé exécutif

Parti d'une base hexagonale, 100 % locale et persistée (Foundry Local, EF Core + SQLite, un snapshot d'état par page), Narratum a d'abord absorbé les quatre features narratives de StoryMaker — choix multiples, personnages, secrets, image par page — sans renoncer à sa propreté. Un provider cloud Azure AI Foundry (auth Entra) est venu compléter l'inférence locale, sélectionnable modèle par modèle.

Le chantier suivant fut l'interface : après un aller-retour Fluent UI (v4 → v5 RC), le choix a été fait d'abandonner Fluent au profit d'un thème éditorial custom (serif Newsreader + sans Hanken Grotesk, accent sienna, papier crème, mode jour/nuit). Les six écrans ont été reconstruits en HTML natif + une feuille de style unique, et la dépendance Fluent entièrement retirée.

✦

Où en est-on aujourd'hui

L'application est fonctionnelle de bout en bout — création, génération multi-agents streamée, lecture, bibliothèque — dans une IHM cohérente et sans dette Fluent. Les évolutions ci-dessous sont des raffinements d'usage, nés de la pratique réelle de l'outil.

Le pipeline de génération

Une page se construit en streaming (le Narrateur écrit la prose affichée), puis une série d'agents spécialisés enrichissent l'état après sauvegarde. C'est cette chaîne que les évolutions viennent raffiner.

04

Évolutions demandées

Six raffinements issus de l'usage réel. Chacun avec son approche, les fichiers concernés et un ordre de grandeur d'effort.

E1

Régénérer l'illustration d'une page

LivréEffort · M

Relancer manuellement la génération d'image quand elle a échoué (ou déçu) sur une page.

Aujourd'hui

L'image est produite dans la passe d'agents post-sauvegarde. Si elle échoue, la page reste sans illustration (ImagePath nul) et rien ne permet de réessayer.

Cible

Un bouton « ↻ Régénérer l'illustration » sous l'image (ou « Générer une illustration » si absente) rejoue l'agent image pour cette page, avec le modèle d'image choisi.

Mise en œuvre
  • Nouvelle méthode IGenerationService.RegeneratePageImageAsync(slot, pageIndex, imageModel).
  • Réutiliser le ImagePrompt déjà persité (colonne existante) ; le régénérer via l'agent ImagePrompt s'il manque.
  • Sauver via ImageStorageService + mise à jour du PageSnapshot (ImagePath).
  • UI : bouton sur la page courante de Generation/Index.razor, avec état « en cours » (spinner).
◆Débloque les pages sans image sans devoir régénérer tout le texte. Fondation pour « régénérer / éditer la page » (voir E-bis).
E2

Lire une histoire depuis l'accueil

LivréEffort · S

Ajouter une action « Lire » sur les cartes du tableau de bord.

Aujourd'hui

La carte d'accueil est un seul lien → /generation (« Reprendre »). Pour lire, il faut passer par la Bibliothèque.

Cible

Deux actions par carte, comme en Bibliothèque : Reprendre → génération, Lire → /reader/{slot}.

Mise en œuvre
  • Dashboard/Index.razor : transformer le pied de carte en deux boutons (Reprendre / Lire) plutôt qu'un lien global.
  • Réutiliser les classes .btn existantes ; aucune logique nouvelle.
◆Quick win pur — quelques lignes de markup.
E3

Première action dans le wizard → génère la 1ʳᵉ page

LivréEffort · M

Une étape pour décrire la première action, afin de générer directement la page 1 à la création.

Aujourd'hui

Le wizard crée l'histoire puis atterrit sur l'écran de génération, où l'utilisateur doit saisir une première intention pour lancer la page 1.

Cible

Une étape « Première action » (avant le récap) ; à la création, l'intention est utilisée pour générer la page 1 immédiatement.

Mise en œuvre
  • Wizard : nouveau champ firstAction + une étape (ou un bloc dans le récap).
  • Après CreateStoryAsync, naviguer vers /generation/{slot} en passant l'intention, et lancer la génération automatiquement au chargement (réutilise l'UI de streaming existante).
  • Alternative — générer dans le wizard : plus lourd (dupliquer l'affichage du streaming). Décision à trancher (§07).
◆Réduit la friction « histoire créée mais vide » : on arrive sur une première page déjà écrite.
E4

Masquer l'agent Narrateur dans les coulisses

LivréEffort · S

Le Narrateur produit la prose déjà affichée : son bloc est redondant dans la modale des coulisses.

Aujourd'hui

La modale liste toutes les traces, dont « Narrateur » — dont le contenu est exactement le texte de la page.

Cible

Filtrer le Narrateur de l'affichage des coulisses (les autres agents restent : Résumé, Cohérence, Personnages, Choix, Secrets, Image).

Mise en œuvre
  • Generation/Index.razor : _agentTraces.Where(t => !EstNarrateur(t)) à l'affichage (on garde la donnée brute).
  • Filtrer sur un identifiant stable de l'agent Narrateur (nom/rôle) — à confirmer côté GenerationService.
◆Coulisses plus lisibles — on n'y voit que la « valeur ajoutée » invisible au lecteur.
E5

Supprimer la page Configuration

LivréEffort · S

Tout se configure désormais au fil de l'eau (modèle par histoire et par page, souscription, image) : la page Réglages n'a plus lieu d'être.

Aujourd'hui

/config gère un modèle de narration « par défaut de session » + un test de connexion LLM, accessible via l'engrenage ⚙ du header.

Cible

Supprimer /config et retirer l'engrenage. ModelSelectionService reste (fournit le défaut) mais sans écran dédié.

Mise en œuvre
  • Supprimer Config/Index.razor ; retirer le lien ⚙ de MainLayout.
  • Le seul élément à valeur réelle est le test de connexion LLM. Options : (a) l'abandonner ; (b) le garder sous forme d'un petit indicateur d'état (pastille verte/rouge) dans le header ou le composer. Décision à trancher (§07).
◆Moins de surface, header épuré. Recommandation : supprimer, et n'ajouter un diagnostic que si le besoin se fait sentir.
E6

Régénérer depuis une page antérieure = tronquer la suite

LivréEffort · M

Revenir sur une page précédente et relancer la génération doit supprimer les pages suivantes (devenues une branche caduque) — après avertissement.

Aujourd'hui

La génération enchaîne depuis la pointe du récit. Régénérer en étant revenu sur une page antérieure n'a pas de sémantique claire, et les pages suivantes resteraient incohérentes.

Cible

Générer (ou choisir une suite) depuis la page N alors que N < dernière page ⇒ modale d'avertissement « les pages N+1 à M seront supprimées », puis troncature et nouvelle génération à partir de l'état de la page N.

Mise en œuvre
  • Déclencheur : _currentPageIndex < _pageHistory.Max() au moment de « Générer la suite » ou de cliquer un choix.
  • Modale de confirmation destructive (réutilise le pattern .overlay/.modal de la suppression d'histoire).
  • Nouvelle méthode de persistance TruncatePagesAfterAsync(slot, pageIndex) puis génération classique — l'archi snapshot-par-page rend la reprise depuis N propre (l'état caché des secrets, le casting, etc. repartent du snapshot de N).
  • Rafraîchir timeline + sidebar après troncature.
◆Transforme la timeline en vrais points de branche : on peut réécrire l'histoire à partir de n'importe quelle page. Action irréversible ⇒ l'avertissement est indispensable.
05

Idées complémentaires

Quelques pistes qui prolongent naturellement l'usage — à prendre ou à laisser.

C1

Exporter l'histoire

Markdown livréEffort · M

Compiler toutes les pages (prose + illustrations) en un fichier téléchargeable — Markdown d'abord, puis EPUB/PDF.

Une maquette antérieure prévoyait un bouton « Export ». Très fort ROI pour un générateur d'histoires : on repart avec un livre. Markdown = trivial ; EPUB/PDF = une lib.

C2

Régénérer / éditer la page courante

Effort · M

Rejouer la génération de la page affichée (autre modèle, intention ajustée) ou corriger la prose à la main.

Symétrique de E1 côté texte. Utile quand une page part de travers, sans repartir de zéro.

C3

« Continuer seul »

LivréEffort · S

Le composer promet déjà « ou laissez le récit continuer seul » — mais aucun bouton ne le fait.

Un bouton qui génère la page suivante sans intention explicite (l'IA enchaîne), ou choisit une des 3 suites automatiquement.

C4

Petits plus

Effort · S
  • Annuler la dernière page — rollback quand une génération rate.
  • Secrets cachés en mode Expert — ils sont stockés ; les montrer à l'auteur (pas au lecteur).
  • Renommer une histoire — éditer le titre depuis la bibliothèque.
  • Régénérer les choix — si les 3 suites n'inspirent pas.

07

Décisions à confirmer

E3 — Où génère-t-on la première page ?

Recommandation : naviguer vers l'écran de génération avec l'intention pré-remplie et lancer automatiquement le streaming. Réutilise l'UI existante, effort moindre. (Alternative : afficher la génération dans le wizard — plus lourd.)

E5 — Garde-t-on un diagnostic de connexion ?

Recommandation : supprimer /config et l'engrenage ; n'ajouter un indicateur d'état LLM (pastille verte/rouge) que si le besoin apparaît. À trancher : suppression totale, ou mini-indicateur conservé ?

E6 — Que déclenche la réécriture depuis une page antérieure ?

Recommandation : toute génération depuis une page N < dernière (bouton « Générer » ou clic sur un choix) ouvre l'avertissement de troncature. À trancher : ce comportement implicite, ou réserver la réécriture à une action explicite « Réécrire à partir d'ici » ?

C1 — Formats d'export prioritaires ?

Markdown est quasi gratuit et suffit pour une v1. EPUB/PDF demandent une librairie. Ordre suggéré : Markdown → EPUB → PDF.

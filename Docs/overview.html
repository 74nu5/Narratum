Narratum

Un moteur narratif déterministe qui garde la cohérence d'une histoire — le LLM n'écrit que la prose.

.NET 10 / C# Architecture hexagonale Blazor Server + FluentUI EF Core · SQLite Foundry Local (LLM privé) 15 projets
01

La thèse

Narratum n'est pas un wrapper autour d'un LLM. C'est un moteur d'histoire : les événements, l'état, la mémoire et l'orchestration garantissent la cohérence narrative de façon déterministe. Le modèle de langage est relégué à ce qu'il fait de mieux — tourner la phrase — et le système fonctionnerait encore, en version mockée, même avec un LLM « stupide ».

Conséquence directe sur le design : tout a été bâti par phases strictes, sans IA d'abord. Le moteur devait être « béton » avant qu'une seule ligne ne soit générée. C'est ce qui explique le découpage en couches ci-dessous, et pourquoi la persistance, la mémoire et les règles existent avant même la couche LLM.

Aucun LLM ne doit écrire une ligne tant que le moteur narratif n'est pas béton. Nous construisons un moteur, pas une démo. Principe directeur — ROADMAP.md

Le pipeline de génération

Ce qui se passe quand on demande la page suivante d'une histoire. Le chemin streaming (nouveau) appelle directement l'agent Narrateur pour afficher le texte en direct, puis persiste la page complète.

04

Fonctionnalités, par phase

Les six phases suivent l'approche « anti-bidouille » : chacune ne démarre qu'une fois la précédente solide. Toutes sont aujourd'hui complètes.

1

Fondations — sans IA Complète

Moteur testable sans LLM : entités, état immuable, event-sourcing, persistance SQLite déterministe, démo CLI (Playground).

2

Mémoire hiérarchique Complète

Quatre niveaux (Événement → Chapitre → Arc → Monde), extraction de faits, état canonique et validation de cohérence.

3

Orchestration multi-agents Complète

Pipeline (Contexte → Prompt → Agents → Validation → Intégration), 4 agents spécialisés (Narrateur, Personnage, Résumé, Cohérence), retry, logging & métriques.

4

Intégration LLM Complète

Foundry Local 100 % local & privé via IChatClient, initialisation paresseuse, retry avec backoff, routage de modèle par agent.

5

Narration contrôlée Complète

Prompts optimisés, cache de résumés, compression de contexte, température par agent — et désormais streaming token-par-token à l'écran.

6

Interface web Complète

Blazor Server + FluentUI : wizard de création, dashboard, lecteur, génération, plus timeline, éditeur et visualisations.

06

Évolutions possibles

Aucune n'est bloquante — ce sont des pistes ordonnées par rapport qualité/effort, ancrées dans l'état réel du code.

Profondeur narrative

  • Activer le multi-agents côté Web : brancher Personnage, Résumé et Cohérence (déjà écrits) dans la génération.
  • Câbler la mémoire (IMemoryService) et la validation de cohérence dans le service Web.
  • Concrétiser Rules : le projet est un scaffold ; y porter un vrai moteur de règles narratives.

LLM & qualité

  • Streaming dans le pipeline complet : aujourd'hui le chemin streaming contourne la validation.
  • Multi-provider : l'adaptateur IChatClient ouvre déjà la porte à Ollama / OpenAI en plus de Foundry.
  • Score de qualité narrative et A/B de prompts pour exploiter la compression de contexte déjà en place.

Expérience & UX

  • Connecter timeline / éditeur / visualisations (composants Phase 6) aux vraies données.
  • Édition inline et régénération de passages, avec le streaming en direct.
  • Export d'histoires (Markdown, ePub/PDF).

Robustesse & ops

  • Migrations EF à la place de EnsureCreated, + sortir la base de démo du versionning.
  • Llm.Tests cross-platform (épinglé win-x64) et E2E Playwright en CI.
  • Observabilité : exposer les métriques de pipeline déjà collectées.
